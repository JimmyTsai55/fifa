01 · Overview

專案總覽

它是什麼
一個回答「2026 世界盃」相關問題的 REST API：陣容、賽程、球星、看點。

怎麼做到
使用者問題 → 分流代理 判斷意圖 → handoff 給 4 個專家代理 → 呼叫工具拿真實足球資料 → 由 LLM 統整自然語言回答。

設計重點
嚴守 Clean Architecture（Ports & Adapters）、外部 API 免費額度下的快取/限流、無金鑰自動部署。

03 · Request Lifecycle

一個問題的端到端旅程

  1. POST /ask 帶 AskRequest{question}
  2. deps.require_api_key 驗證 X-API-Key（缺/錯 → 401）
  3. container.build_use_case() 組裝代理、協定實作、配額服務
  4. AnswerQuestionUseCase.execute() → QAService.answer()（anyio 轉非同步）
  5. Runner.run_sync(triage, question) 啟動代理
  6. Triage 分流 → handoff 給 4 專家之一
  7. 專家執行 function tools → 觸發 FootballDataProtocol
  8. Adapter：查快取(TTL) → 過期才打 API → 限流(10/分) → 記配額
  9. LLM 統整 → final_output
  10. 200 回 AskResponse{answer, model}

錯誤映射：401 驗證失敗 · 422 輸入無效 · 429 配額用盡 · 502 上游 API 錯誤

04 · AI 多代理設計（三）

Function Tools 與依賴注入

# adapters/agent_tools.py
@function_tool(failure_error_function=None)
def tool_find_team(name: str) -> int:
    """球隊拉丁名 → team_id；查無回 0。"""
    return current_football().find_team_id(name) or 0
# core/container.py — 啟動時綁定實作
agent_tools.bind_football(football)
#   ↑ 注入 FootballDataProtocol

為什麼這樣設計

  • 工具只依賴 協定，不認得 SQLite / httpx
  • 真正的資料來源（ApiFootballAdapter）在組裝根注入
  • 測試時可注入 mock，65 個測試全程不需金鑰

編排核心

# openai_agents_adapter.py
def answer(self, q: str) -> str:
    return Runner.run_sync(
        self._agent, q
    ).final_output

06 · Database

資料庫設計：SQLite + WAL

-- db/schema.py
CREATE TABLE cache_meta (
  cache_key    TEXT PRIMARY KEY,
  fetched_at   REAL NOT NULL,   -- Unix 時間戳
  payload_json TEXT NOT NULL    -- API 回應 JSON
);

CREATE TABLE kv (
  key   TEXT PRIMARY KEY,
  value TEXT NOT NULL           -- 配額狀態等
);

設計取捨

  • 只有兩張表：API 回應整包以 JSON blob 存放，key 如 squad:1
  • WAL 模式：支援併發讀（concurrency 50）
  • RLock：執行緒安全存取快取/配額
  • 無 ORM、無 migration：原生 sqlite3

適配器 sqlite_store_adapter.py 同時實作 CacheStoreProtocol 與 QuotaStoreProtocol，可替換後端。

class RateLimiter:
    def __init__(self, max_per_min=10):
        self._hits = deque()
    def allow(self) -> bool:
        # 清掉 60s 前的紀錄，<10 才放行
        return len(self._hits) < self.max
  • 每日配額：讀 x-ratelimit-requests-remaining 標頭，存進 kv

async function submit() {
  const r = await fetch("/chat", {
    method: "POST",

10 · CI/CD（一）

持續整合：GitHub Actions

# .github/workflows/ci.yml

10 · CI/CD（二）

持續部署：WIF + Cloud Run

# deploy.yml（push main，test 通過後）
permissions:
  id-token: write   # ← WIF 關鍵
- uses: google-github-actions/auth@v2
  with:
    workload_identity_provider: ${{ secrets.GCP_WIF_PROVIDER }}
    service_account: ${{ secrets.GCP_SERVICE_ACCOUNT }}
- run: gcloud run deploy fifa-wc-agent ...

Workload Identity Federation：GitHub 以 OIDC 換取 GCP 短期憑證，不存任何長期金鑰。

關鍵部署參數

  • --source .：由原始碼自動建置（Buildpacks）
  • --max-instances 1：單實例，確保每日配額計數正確
  • --update-secrets：從 Secret Manager 注入 4 把金鑰
  • --concurrency 50 · --timeout 180s
  • --no-allow-unauthenticated：私有服務，IAM 層擋匿名請求
  • 煙霧測試：匿名 curl / 預期 403（接受 2xx/401/403=已上線且鎖定生效）

concurrency: deploy-cloud-run 群組避免重疊部署。
